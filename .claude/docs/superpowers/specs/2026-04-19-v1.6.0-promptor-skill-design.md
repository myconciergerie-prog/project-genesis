# v1.6.0 — Promptor skill (MINOR)

## Summary

Promote the Promptor pépite — a canonical system prompt template calibrated for Claude Opus 4.7 1M + MCP protocol prompt engineering — into a standalone Genesis skill at `skills/promptor/`. Establish the skill as the single source of truth; demote the existing Layer 0 pépite to a synced cache role. Add the 8th Genesis skill with privilege class `none`. Orthogonal to Layer A/B (meta-tool, not part of the 7-phase bootstrap).

## Motivation

The Promptor template was captured 2026-04-18 as a Layer 0 pépite (`~/.claude/memory/layer0/pepite_promptor_template_anthropic_prompt_engineering.md`). The binding rule (`~/.claude/memory/layer0/feedback_invoke_promptor_for_production_anthropic_prompts.md`) made it actually used rather than rotting in latent memory. Together they cover the user's machine but **do not propagate to other machines** or to **new projects bootstrapped via Genesis**.

The cross-project pain is real: every Anthropic-API-using project bootstrapped via Genesis (Aurum.ai runtime agents, Cyrano AI assistant, Myconciergerie ride-booking agent, plus future projects) needs production-grade prompt engineering. Without a Genesis-shipped skill, each new project would either (a) re-derive the template manually, (b) copy-paste the pépite from this user's Layer 0, or (c) skip the discipline and ship sub-calibrated prompts.

The 2026-04-19 Option C decision (recorded in v1.5.0 session context and re-confirmed in v1.5.1 resume) flagged Promptor skill promotion as the v1.6.0 target after v1.5.x line completion. v1.5.1 closed v1.5.0's three-gap honesty correction. v1.6.0 opens this new track.

## Non-goals

- **Not a v2 conversational pipeline.** The "Promptor" name appears in two distinct Genesis design tracks: (1) this v1.6.0 skill = system-prompt template for engineers (the pépite), and (2) the v2 conversational drag-drop → extract → formalize pipeline (`.claude/docs/superpowers/specs/v2_vision_promptor_fusion.md`). They share a name and Anthropic-ecosystem orientation; they are not the same artifact. This spec covers only (1).

  **Forward-naming reservation** : the `skills/promptor/` namespace is hereby committed to the **template** track (the pépite), because (a) the pépite's `<role>` literally declares `Tu es "Promptor"` — Promptor proper is the template artifact, and (b) the v2 spec already uses the looser phrasing "Promptor-inspired conversational layer" (line 14 verbatim), framing v2 as inspired by Promptor rather than being Promptor itself. When the v2 conversational surface ships, it will land under a different skill name — provisional candidates: `genesis-conversational-architect`, `genesis-spark-and-mirror`, or one skill per Étape mirroring the existing `genesis-drop-zone` pattern (`genesis-spark` for Étape 1, `genesis-mirror` for Étape 2, `genesis-iterate` for Étape 3). The v2 spec's current narrative uses of "Promptor" should be re-read as "Promptor-inspired" once v2 lands, not as a brand commitment for v2's skill names. This reservation is recorded here so v2 ship work has the naming-collision resolution baked in.
- **No runtime test harness.** Skill auto-discovery via description match is exercised by Claude Code's skill engine at session level, not by in-skill tests. Validation in this ship = paper-trace (grep for template integrity, frontmatter description match against binding rule triggers, plugin.json parse, install dry-run = no project file plant).
- **No domain-adapted examples shipped.** Adaptation guidance is documented (in `references/template.md` § "Adaptation points") but no pre-cooked examples (data-engineer, security-researcher, legal-drafter) are bundled. Rationale: examples decay faster than principles, and pre-cooking risks copy-without-calibration.
- **No `/promptor` slash command.** All 7 existing Genesis skills surface via skill-description match (no `commands/` dir in plugin). Adding a slash command for Promptor only would be a structural anomaly. Direct invocation remains available via the `Skill` tool.
- **No automated Layer 0 pépite sync.** The pépite update on ship is a one-time manual edit (header note "Synced from skill v1.6.0 @ 2026-04-19"). No script, no hook. If the pépite drifts or the skill evolves, the next sync is also manual. Anti-Frankenstein: a sync script earns its keep only when sync events happen frequently enough to justify the maintenance cost.

## Architecture

### File layout

```
skills/promptor/
├── SKILL.md                 (~60 lines)
└── references/
    └── template.md          (~80 lines)
```

**No `install-manifest.yaml`.** Promptor writes nothing to the host project (consistent with the 2 existing privilege-`none`/no-disk-side skills: `genesis-drop-zone`, `phase-5-5-auth-preflight`). Promptor will be the 3rd skill without an install-manifest.

### `SKILL.md` — invocation gate

Frontmatter description carries the **full trigger list verbatim** from the binding rule (so Claude Code's auto-discovery surfaces the skill on the same triggers the binding rule names). Body covers:

1. **What this skill is** — one-paragraph identity. Cross-link the canonical source (the 1:1 mirror declaration: "1:1 mirror of `~/.claude/memory/layer0/pepite_promptor_template_anthropic_prompt_engineering.md`. If pépite content updates, this skill must be re-synced. Last sync: v1.6.0 @ 2026-04-19").
2. **When to invoke** — explicit triggers (FR + EN) + implicit triggers (about to write/modify any LLM prompt in an Anthropic-first project).
3. **When NOT to invoke** — non-technical user, trivial one-shot, exploratory phase, non-Anthropic context. Mirror pépite § "Quand NE PAS utiliser ce template".
4. **Two-phase operating mode (high-level)** — Phase 1 standby (acquire specs + halt) / Phase 2 creation (render Parts A/B/C/D). The verbatim XML lives in `references/template.md` to keep `SKILL.md` tight on **when** vs **what**.
5. **Adaptation discipline** — what to swap (`<role>`, `<output_structure>` modules), what to keep (`<phase_*>` gates, `<global_constraints>`). Mirror pépite § "Adaptation points".
6. **Source-of-truth declaration** — explicit "this skill is canonical; Layer 0 pépite is a synced cache for sessions outside Genesis-bootstrapped projects".

### `references/template.md` — canonical XML

Holds the **verbatim XML template** + the 6 architectural principles + adaptation points + when-not-to-use + cross-project utility list. Mirror of pépite § "Template canonique — verbatim", § "Architectural principles", § "Quand NE PAS utiliser ce template", § "Adaptation points", § "Cross-project utility".

Read by Claude in Phase 2 (when actually rendering a prompt). Phase 1 uses `SKILL.md` only — minimizes tokens read on gate decisions.

### Privilege class

`none` — no disk write (no project files, no archive), no network (no external API calls), no subprocess (pure prompt-template rendering by Claude). 8th skill in the concentrated-privilege map. Joins `journal-system` as the **2nd** privilege-`none` skill (the others per `master.md` pattern #2 verbatim: phase-minus-one = subprocess via installers, phase-5-5-auth-preflight = subprocess via SSH/PAT/repo creation, session-post-processor = disk via redacted archives, pepite-flagging = disk via cross-project pointer files, genesis-protocol = disk via project-directory writes, genesis-drop-zone = disk + network multi-class).

### Layer assignment

**Orthogonal to Layer A / Layer B.** Promptor is a meta-tool invocable cross-session, cross-project, outside the bootstrap protocol. It is not part of Étape 0/1/2/3 (Layer A) nor Phase -1/0/1/…/7 (Layer B). It is a peer-tool for Claude calling cross-cutting prompt-engineering discipline on demand.

Per anti-Frankenstein: do not introduce a third "Layer M" (meta) category for one skill. Document Promptor as orthogonal in `master.md` and let the layer category emerge naturally if a 2nd meta-skill arrives.

### 1:1 mirror discipline (cross-skill-pattern #1)

`SKILL.md` + `references/template.md` together = byte-for-byte content mirror of the Layer 0 pépite (split into 2 files for the gate/template separation; semantic content unchanged). Drift = merge-blocker. Header in `SKILL.md` declares: "1:1 mirror of `~/.claude/memory/layer0/pepite_promptor_template_anthropic_prompt_engineering.md`. If pépite content updates, this skill must be re-synced. Last sync: v1.6.0 @ 2026-04-19."

This is the **third** skill applying pattern #1 (after `journal-system` mirroring Layer 0 journal spec and `pepite-flagging` mirroring `specs/v1_pepite_discovery_flagging.md`). With three data-points, the pattern is now stable.

### Cross-skill-pattern #2 (concentrated privilege map) data-point

8th skill, privilege class `none`, no mitigations needed (no privilege to mitigate). Joins `journal-system` (none) as the 4th privilege-`none` skill — but unlike journal-system which has no privilege because it operates inside the user's own memory directory with consent already gated by trigger phrases, Promptor has no privilege because it has nothing to do with disk/network/subprocess at all. This is the cleanest possible map entry.

### Cross-skill-pattern #4 (Layer A/B + zero-ripple) data-point

Promptor is orthogonal to Layer A/B → no ripple to declare on the bootstrap pipeline. Zero edits to Layer A skills (`genesis-drop-zone`), zero edits to Layer B skills (`genesis-protocol`, `phase-minus-one`, `phase-5-5-auth-preflight`, `journal-system`, `session-post-processor`, `pepite-flagging`). The only cross-cutting touch is `master.md` (narrative update) and `skills/README.md` (skill list addition). This counts as the **seventh data-point of zero-ripple** — extending the principle from "Layer A grows / corrects / opt-in renders without rippling Layer B" to "new orthogonal meta-skill ships without rippling either layer".

## Acceptance criteria

A v1.6.0 PR is merge-ready when all of the following hold:

1. **Files exist** : `skills/promptor/SKILL.md` and `skills/promptor/references/template.md` exist with SPDX headers.
2. **Frontmatter parseable** : `SKILL.md` frontmatter has `name: promptor` + a `description` field that contains all 4 explicit trigger phrases from the binding rule **verbatim** (exact string match — no paraphrase, no abbreviation) + the 1 implicit trigger summary. Verified by `grep -F` against the binding rule's exact phrasing.
3. **Template integrity** : `grep "Promptor" skills/promptor/references/template.md` returns the verbatim `<role>` block from the pépite. Byte-for-byte XML structure preserved (5 sections: `<role>`, `<operating_mode>` with `<phase_1_standby>` + `<phase_2_creation>` + `<output_structure>`, `<global_constraints>`).
4. **No install-manifest** : `ls skills/promptor/install-manifest.yaml` returns no such file (intentional).
5. **Skill is read-only** : `skills/promptor/` contains no `install-manifest.yaml`, no script that writes outside `skills/promptor/`, and no scaffold templates intended for downstream-project planting. Verified by `grep -rE "(Write|cp |mkdir|touch|>) " skills/promptor/` returning zero project-write candidates (any false positives — e.g. `cp` inside an XML example — must be in a verbatim quoted XML block).
6. **Plugin.json valid** : `cat .claude-plugin/plugin.json | python -m json.tool` parses. Version field = `1.6.0`. `keywords` array contains `prompt-engineering` and `promptor`.
7. **Layer 0 pépite header note present + § Cross-references tense fixed** : (a) pépite file at `~/.claude/memory/layer0/pepite_promptor_template_anthropic_prompt_engineering.md` contains the synced-cache header note immediately after frontmatter, AND (b) the existing § "Cross-references" line "**Genesis v1.6.0 candidate**" is replaced by the past-tense fact "**Genesis v1.6.0 promotion** (2026-04-19) — promoted to `skills/promptor/`. This Layer 0 entry is now a synced cache."
8. **Binding rule update present** : feedback file mentions `Skill` tool invocation as preferred path when skill installed.
9. **CHANGELOG entry honest** : 5-axis self-rating present, mean ≥ 9.0, anti-Frankenstein deduction (or absence thereof) explicitly priced.
10. **Resume file written** : next-session resume exists with PowerShell launcher one-liner per Layer 0 `feedback_end_of_session_resume_phrase` discipline.
11. **PR squash-merged + tag** : branch `feat/v1.6.0-promptor-skill` PR'd against main, squash-merged, tag `v1.6.0` pushed.

## Anti-patterns avoided

- **Pre-cooked domain examples** (rejected): would invite copy-without-calibration. Examples grow stale; principles don't.
- **Slash command `/promptor`** (rejected): structural anomaly vs 7 existing skills, marginal UX gain.
- **Layer M (meta) category for one skill** (deferred): introduce only when ≥2 meta-skills exist.
- **Auto-sync script for pépite** (rejected): manual ship-time sync is sufficient at expected frequency.
- **Bundling spec for v2 conversational Promptor** (rejected): different artifact, different track. Cross-link only.
- **Retiring the Layer 0 pépite outright** (rejected — Q1 Option B): would break sessions opened outside Genesis-bootstrapped projects (the user has many such sessions on this machine).
- **Pépite as canonical, skill as copy** (rejected — Q1 Option A): would leave the marketplace-shipped form vulnerable to per-machine drift, with no clear authority for new-project-bootstrap inheritance.

## Out-of-scope follow-ups (for v1.6.1 or later)

- **Runtime auto-discovery validation** — spawn a fresh Claude Code session in any Genesis-bootstrapped project, type one of the trigger phrases, observe the skill engine surfacing `promptor`. Cannot be done in this session because skill is not yet installed at session-open. v1.6.1 candidate.
- **Domain adaptation examples reference** — if user reports re-deriving the same adaptation pattern multiple times across projects, capture as a `references/adaptations/<domain>.md` file. Pain-driven, not preemptive.
- **Auto-sync script** — if Layer 0 pépite ↔ skill drift becomes a real problem (observed > 0 times), write a `scripts/sync_layer0_pepite.sh` that diffs and warns. Pain-driven.
- **Promptor v2 conversational pipeline** — separate v2 spec already exists (`.claude/docs/superpowers/specs/v2_vision_promptor_fusion.md`). Independent track. Forward-naming reservation for v2's skill name(s) is recorded in Non-goal #1.
- **Cross-link skill to v2 vision spec** — once v2 lands, add a `## Related skills` section to `SKILL.md` pointing at the conversational Promptor surface (under whatever v2 skill name it ships with). Deferred until v2 exists.
- **Sync-timestamp drift maintenance vector** — the literal "Last sync: v1.6.0 @ 2026-04-19" string appears in three places (SKILL.md header, pépite header note, optionally referenced in binding rule). A v1.6.x patch that touches the template will need to update all three by hand. **Pain-trigger for the rejected auto-sync script reconsideration: the first observed drift event** (e.g., a v1.6.x release where SKILL.md is updated but the pépite header note is missed). Until then, the manual convention is acceptable.
- **Anthropic prompt-engineering 2026-04 SOTA grounding** — see Best-at-date axis honest deduction. v1.6.1 candidate to commission a fresh `sota/anthropic-prompt-engineering_<date>.md` R8 entry and re-anchor the template's choices.
- **Cross-device Layer 0 sync (multidevice scenario)** — H7 refuted by construction; the v1.6.0 binding rule update is single-machine. Future Layer 0 hygiene work (cross-device propagation discipline) is its own thread, separate from v1.6.x line.
