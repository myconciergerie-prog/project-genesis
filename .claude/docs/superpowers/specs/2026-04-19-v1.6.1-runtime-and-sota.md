# v1.6.1 Spec — Runtime auto-discovery + Anthropic prompt-eng SOTA grounding

## Summary

v1.6.0 shipped the Promptor skill at honest 9.02/10 with two axis caps priced in at spec time :

- **Pain-driven 9.2** — H1 (skill auto-discovery surfaces `promptor` on trigger phrases) confirmed only by paper-trace (frontmatter `description` verbatim grep), not runtime observation.

v1.6.1 closes both caps in one ship. Evidence captured as transcript excerpt + structured runbook under `tests/`.

Honest rating projection post-v1.6.1 : **9.20 ± 0.05**, third consecutive streak ≥ 9.0.

## Why

Deferred-but-inevitable closure of priced-in v1.6.0 debt. The "close the current minor line before stacking new features" discipline is the same reflex that made v1.5.1 → v1.5.0 honesty correction work. Skipping straight to v1.7.0 (or v1.5.2 runtime dogfood of a different skill) would accept v1.6.0 with known axis debt — exactly the v1.5.0 mistake pattern that broke the 11-ship streak.

Two axis caps closing in one ship is more efficient than two separate ships. SOTA commissioning is blocking for the template patches (can't ship "SOTA-grounded edits" without the SOTA being shipped first) ; runtime validation is blocking for the binding-rule pinning decision (can't resolve `promptor` bare vs `genesis-protocol:promptor` namespaced without empirical evidence).

## Three brainstorming questions resolved

**Q1 — Patch scope: Option A (P1 only) / Option B (P1 + P2) / Option C (P1 + P2 + structured-outputs section for Part B)** → **Option B selected**.

Rationale : Option A leaves known low-risk polish items unaddressed, creating PATCH debt. Option C re-opens the Anthropic structured-outputs vs free-form XML decision (R8 open question 5) without empirical grounding on which is better for Promptor's typical use cases (agent personas vs data extraction). Defer to pain-driven v1.6.2. Option B ships all SOTA-confirmed surgical edits in one PATCH and stops there.

**Q2 — Runtime evidence capture: Option A (screenshot + transcript paste to spec) / Option B (structured runbook at `tests/` + evidence log) / Option C (journal entry only)** → **Option B selected**.

Rationale : Option A couples evidence to the spec file, which becomes stale once merged. Option C puts validation evidence in journal but makes it un-rediscoverable for future v1.6.x dogfood cycles. Option B creates a reusable runbook (future v1.6.x releases repeat the validation cheaply) + versioned evidence trail. Aligns with v1.5.1 dogfood-first discipline (evidence files under `memory/project/` or `tests/`).

**Q3 — Binding-rule pinning approach: Option A (pin to `project-genesis:promptor` proactively) / Option B (keep non-pinned until evidence forces change) / Option C (pin and provide non-pinned as back-compat)** → **Defer to runtime evidence** (resolved at AC#7).

Rationale : Pinning is a Layer 0 surface change (universal impact on this machine). Making the change without evidence risks friction across all future Anthropic-API sessions. Non-pinned status from v1.6.0 is explicitly "awaiting empirical validation" per that spec's Section "forward-naming reservation". AC#7 routes decision per observed runtime behavior, respecting the evidence-before-prose discipline.
