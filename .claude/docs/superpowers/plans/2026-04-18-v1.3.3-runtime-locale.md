## Design decisions

### Why v1.3.3 closes runtime locale detection now, not later

v1.3.0 / v1.3.1 / v1.3.2 each authored EN variants day-1 per R9 tier 3. All seven EN templates exist on disk since v1.3.2 ship. The gap is a one-variable dispatch per stage — not a template creation effort. EN-native users have been seeing FR-only welcome and FR-only mirror since v1.3.0; closing the loop now turns a quiet R9 violation into zero-cost R9 closure. Deferring further would waste already-authored content.

### Why two locale variables, not one

Welcome precedes any content. Mirror-onwards is content-driven. A single variable forces one of:
- **Always-FR welcome** — wastes the reliable signal of an EN intent-match phrase. The user who types `I want to create a project` deserves an EN welcome, not an FR box.
- **Block welcome behind content** — shatters the v1.3.0 "welcome box IS the invitation" UX principle. Loses the pre-content opening moment.

Two variables with distinct lifecycles (`welcome_locale` at invocation, `content_locale` after first content turn) honour the best signal available at each render point. The cost is documented divergence in edge cases (e.g. EN intent + FR content → EN welcome + FR mirror). This is acceptable: each surface is internally consistent within its lifecycle window.

### Why frontmatter null-class tokens stay FR canonical (not localised)

Layer B's `phase-0-seed-loading.md` Step 0.2a detects null classes by exact string match on `"a trouver ensemble"` / `"non mentionne"` / `"non mentionnee"` / `"a affiner — ..."`. Localising the frontmatter tokens (e.g. EN canonical `"to be found together"`, `"not mentioned"`) would require Layer B to grow a bilingual parsing branch — a v1.4+ scope, not a v1.3.x ship. Keeping frontmatter FR canonical while body renders in detected locale is an intentional and spec-documented asymmetry: **frontmatter = dev-layer data contract parsed by Layer B; body = user-facing locale-detected human echo**. The Layer A / Layer B contract stays 100% backwards-compatible with v1.3.2.

### Why `mixte` → FR tiebreaker

`langue_detectee` admits three values (`FR` / `EN` / `mixte`). Display needs a deterministic tiebreaker for `mixte`. FR is picked because:
- Primary project language on this machine (Layer 0 user profile, French-speaking environment).
- Trigger-list ordering in `SKILL.md` frontmatter starts with FR phrases.
- Matches v1.3.2 hardcoded default — users already habituated to FR.
- "Always bilingual for mixte" would re-fragment the surface — exactly the anti-pattern v1.3.3 removes.

Users can re-state their intent more clearly to shift `langue_detectee` from `mixte` to `FR` or `EN`. No hidden UI behaviour.

### Why slash command → FR default

`/genesis-drop-zone` carries no language signal at invocation. Options:
- **FR default (picked)** — deterministic, aligns with FR-first trigger ordering, consistent with prior versions for users already habituated to slash → FR.
- **Read `$LANG` / `$LC_ALL`** — heavier, error-prone on Windows (can be `C`, unset, or misconfigured), adds environment surface area for tiny benefit. An EN-locale user typing `/genesis-drop-zone` and then dropping EN content sees: FR welcome → EN mirror + EN rest. Graceful degradation at the cost of one FR welcome box. The intent-phrase path is the locale-native path; slash is the engineer path where FR is a defensible default.

### Why PATCH (v1.3.3), not MINOR (v1.4.0)

v1.3.x is the conversational-layer ship line. v1.3.0 opened it with a welcome. v1.3.1 structured the mirror. v1.3.2 wrote the file + wired Layer B. v1.3.3 polishes the rendering surface — same feature line, same surface area, backwards-compatible (a v1.3.2 install renders FR everywhere, a v1.3.3 install renders in detected locale; both read / write drop_zone_intent.md with the same Layer B contract). Not a new thematic line. PATCH is correct.

### Why one new bilingual pair (EN zero-content re-prompt) and not more

Audit of v1.3.0–1.3.2 templates:
- Welcome box FR + EN ✓ (v1.3.0)
- Mirror template FR + EN ✓ (v1.3.1)
- 9-field labels FR + EN ✓ (v1.3.1)
- Consent card FR + EN ✓ (v1.3.2)
- Halt message FR + EN ✓ (v1.3.2)
- Accept bridge FR + EN ✓ (v1.3.2)
- Decline bridge FR + EN ✓ (v1.3.2)
- Context-guard redirect FR + EN ✓ (v1.3.0) — always-bilingual by design, pre-trigger-evaluation
- **Zero-content re-prompt**: only FR was authored in v1.3.0 (ASCII-pure `Je t'écoute — dépose ou écris ce que tu veux me partager.`). This is the only gap. v1.3.3 authors the EN counterpart `I'm listening — drop or write whatever you want to share.`

Narrow scope: the new bilingual pair is the only runtime-string-authoring work in this ship. Everything else is dispatch wiring.

### Target self-rating calibration

Aspirational ≥9.3 on every axis; floor ≥9.0 on every axis. Expected strengths:
- **Pain-driven 9.3+** — closes a concrete R9 tier-3 violation visible since v1.3.0. EN-native users literally see FR UI today; v1.3.3 fixes that end-to-end.
- **Best-at-date 9.2–9.3** — inline R8 citations preserved (`v2_promptor_fusion_landscape_2026-04-17.md` Stage 1 UX patterns still fresh until 2026-04-24).
- **Self-contained 9.3+** — one PR, one skill touched (`genesis-drop-zone`), zero Layer B ripple. Narrower than v1.3.2.
- **Anti-Frankenstein 9.4+** — no new privilege class, no new dependency, no speculative surface. Reuses every existing EN template. One-variable-per-stage dispatch is minimum viable.
- **Prose cleanliness 9.2+** — one spec addition, inline notes on existing sections, no duplication, consistent version-scoped naming.
