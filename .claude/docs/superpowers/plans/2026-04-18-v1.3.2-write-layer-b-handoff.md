## Design decisions

### Why v1.3.2 bundles Layer A write + Layer B integration in one ship

Anti-Frankenstein gate: declare privileges for code that ships with a downstream reader, never speculatively. Shipping the Layer A write without updating Layer B would leave `drop_zone_intent.md` on disk with nothing that reads it. Bundling keeps the cross-layer contract atomic — one ship, one PR, one tag.

### Why rename Layer A file to `drop_zone_intent.md`

`genesis-protocol` already owns `memory/project/bootstrap_intent.md` (Phase 0 Step 0.5 write). Same filename at cwd would be a faux ami: different schema, different path, different audience. Origin-tagged renaming (`drop_zone_intent.md`) makes the Layer B detection logic unambiguous and collides with no user-written filename convention.

### Why narrow privilege — cwd only, no mkdir, halt-on-existing

First Layer A privilege = precedent every future Étape 1/2/3 privilege will be measured against. Start tight:
- **cwd only** — the context guard already asserts cwd fresh; keeping the privilege tight to cwd means the filesystem surface is 1 file.
- **No `mkdir`** — `mkdir` on arbitrary paths inherits Windows `\` vs `/`, UNC, OneDrive, symlink, permission edge cases. Disproportionate dette for a first ship.
- **Halt-on-existing, no overwrite / no timestamp-suffix / no second consent** — the context guard already asserts a fresh cwd. An existing `drop_zone_intent.md` is a genuine anomaly that deserves user reconciliation, not a silent overwrite or ambiguity-creating suffix. Matches the halt-on-leak precedent of `session-post-processor`.

### Why two bridges, not one conditional

Accept and decline are conceptually different voices (action taken vs action refused). A single bridge with a conditional tail would dilute both. Two variants keep each voice honest; the accept bridge instructs (`tape /genesis-protocol`), the decline bridge invites (`relance-moi quand tu veux`).

### Why pure Markdown skill (no Python runtime for the write)

Consistent with every other Genesis skill (`phase-minus-one`, `phase-5-5-auth-preflight`, `journal-system`, `session-post-processor` run.py, `pepite-flagging`, `genesis-protocol`) — the SKILL.md is the dispatch, and Claude uses its native harness tools (Read, Write, Bash, Glob) to execute. The spec's "atomic temp + rename" language describes the conceptual discipline; at runtime Claude uses the `Write` tool (single-operation file creation, harness-atomic for files < 10 KB). For the existence check, Claude uses `Bash` with `test -e` (catches regular files, symlinks, directories alike — matches the spec's `exists()` semantic).

### Why PATCH (v1.3.2), not MINOR (v1.4.0)

v1.3.x is the conversational-layer ship line. v1.3.0 opened it, v1.3.1 structured the mirror. v1.3.2 extends the skill's existing Phase 0 with a write gate + Layer B wire. Backwards-compatible: a v1.3.1-only install still runs correctly, the write is opt-in via consent. Not a new thematic line. PATCH is the correct semver; naming it v1.4.0 would falsely signal a break.

### Why living spec (single file) + one plan file covering two skills

Same rationale as v1.3.1: multiplying specs/plans per skill fragments the mirror discipline. The cross-layer wire is one coherent unit of work — one spec documents the full Étape 0 → Phase 0 flow end-to-end, one plan executes all seven file touches in sequence. Layer B changes are tightly scoped (4 file touches, ~80 lines total) — not large enough to warrant a separate ship.

### Target self-rating calibration

Aspirational ≥9.3 on every axis; floor ≥9.0 on every axis. Best-at-date keeps its inline R8 citations lever (`v2_promptor_fusion_landscape_2026-04-17.md` still fresh until 2026-04-24). Pain-driven strongest: v1.3.2 closes the "creation is coming soon" v1.3.1 bridge promise by writing the intent to disk — direct user-visible delivery. Anti-Frankenstein strong: narrow privilege, halt-on-existing not overwrite, bundled Layer B integration not speculative, no API introduced yet. Prose cleanliness consistent with v1.3.1's bar. Self-contained: one PR ships two skills + privilege map + CHANGELOG + session trace + resume; nothing dangles.
