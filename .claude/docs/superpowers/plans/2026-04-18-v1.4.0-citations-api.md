# v1.4.0 — Citations API Extraction Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship v1.4.0 as a **MINOR increment** on the v1.3.x conversational-layer line — the first MINOR bump since v1.3.0 opened the line. Introduce the **second concentrated privilege class** for `genesis-drop-zone`: an Anthropic API call via Python subprocess (`skills/genesis-drop-zone/scripts/extract_with_citations.py`) that performs Citations-enabled extraction of the 9-field intent schema with per-field source attribution. Mirror renders `[page N]` / `[lines X-Y]` annotations on rows that receive citations; `drop_zone_intent.md` frontmatter extends with optional `<field>_source_citation` nested keys. **Graceful silent fallback** to v1.3.3 in-context extraction when `ANTHROPIC_API_KEY` is unset, Python is unavailable, the `anthropic` SDK is not installed, the API returns an error, or the subprocess output fails schema validation. **Zero Layer B ripple** — additive frontmatter keys only, Layer B Step 0.2a parser ignores unknown keys naturally, no schema_version bump. Six-commit rhythm: spec ✓ + spec polish ✓ + plan (this commit) + plan polish + feat + chore.

**Architecture:** Python subprocess pattern — the SKILL.md dispatch invokes a standalone `extract_with_citations.py` via Bash after the welcome box and initial content turn. Python portability via `$(command -v python || command -v python3 || command -v py)` (same pattern as `session-post-processor/run.py` since v0.6.0). The script reads a single JSON payload on stdin (cwd + attachments[] + typed_text + content_locale_hint + model), calls the Anthropic Messages API with `citations: {enabled: true}` per document block and `cache_control: {type: "ephemeral", ttl: "1h"}`, and emits a single JSON object on stdout carrying the 9-field extraction plus per-field `source_citation` entries. All non-zero exit codes route to the v1.3.3 fallback path. The SKILL.md dispatch layer does the flag evaluation, subprocess launch, JSON parse + schema check, and fallback routing — no new logic in `phase-0-welcome.md` (which keeps being pure template surface).

**Tech Stack:** Claude Code plugin Markdown skills with embedded Python (via Bash dispatch). New dependency: `anthropic` Python SDK (documented in R8 stack entry, pinned at ship time). SPDX headers on every new file. R9 tier-1 English for the Python extractor; user-facing citation annotations are language-neutral ASCII (`[page N]`, `[lines X-Y]`). `gh` CLI with R2.3.1 active-account pre-flight. Semver MINOR bump (plugin.json 1.3.3 → 1.4.0).

## Design decisions

### Why MINOR (v1.4.0), not PATCH (v1.3.4)

Three stacked novelties force the MINOR signal:

1. **Second concentrated privilege class** (network, orthogonal to v1.3.2's disk). Cross-skill-pattern #2 in master.md evolves from "at most one privilege per skill" to "at most one privilege per operation class, per skill". This is a permanent pattern refinement, not a v1.4.0 exception.
2. **First external dependency** in Genesis runtime — `ANTHROPIC_API_KEY` env var + `anthropic` Python SDK package. Previously all skills ran purely on the harness's multimodal capabilities. Adding an external hard dependency (even with graceful fallback) is architecturally different from any prior Genesis ship.
3. **First subprocess invocation in `genesis-drop-zone`** — the skill previously did welcome + mirror + consent + write entirely in-context (v1.3.2's write used the harness `Write` tool, not a Python subprocess). Now the extraction path can leave the harness entirely.

Precedent: v1.3.2 shipped the first Layer A privilege (disk) at PATCH because it was a surface addition within the established conversational-layer line. v1.4.0 does not add to the surface — it changes the substrate. MINOR is the correct tranche. The 0.37 tampon above the 8.5 running-average floor (8.87 pre-v1.4.0) absorbs a MINOR ship with ≥ 8.5 self-rating comfortably.

### Why Python subprocess, not curl or sub-agent

Brainstormed three options at session open:

- **curl via Bash tool** — rejected. PDF base64-encoded payloads exceed Windows/bash command-line length limits (`MAX_PATH`-derived); quoting differs across bash / PowerShell / cmd; `citations_delta` streaming parsing from curl stderr is non-trivial; secret handling via env vars is OK but forensic-log leak risk higher in shell.
- **Python subprocess via `anthropic` SDK** — selected. SDK handles streaming, retries, base64 encoding, `cache_control` serialization, error formatting. Precedent exists: `session-post-processor` ships `run.py` since v0.6.0 using the exact Python portability pattern (`command -v python || command -v python3 || command -v py`). One new runtime dependency (`anthropic`), isolated to one script.
- **Sub-agent via Agent tool** — rejected. Citations is a response-level API artefact (`citations[]` with `cited_text` + `document_index`); Claude Code's Agent tool interface does not expose that metadata to parents. Sub-agent path would degenerate into prompt-based CoVe (soft citations), not API-hard Path A. Does not deliver what the spec specifies.

### Why silent graceful fallback, not a user-facing note

An early draft considered a bilingual note ("Source attribution unavailable this session — extracted in-context") printed before the mirror on the fallback path. Rejected:

- Users without `ANTHROPIC_API_KEY` would see the note every session — adds R9 noise.
- The note leaks implementation detail (API key presence) into a Victor-facing surface.
- The note would need its own bilingual pair (v1.4.0's R9 tier-3 surface area grows).
- Citations are additive value; their absence should be invisible at the surface layer.

Instead: fallback fires silently. The mirror renders v1.3.3-identical output (no `[page N]` annotations). Forensic state goes to subprocess stderr where developers can inspect during integration testing and cost-trending.

### Why additive frontmatter keys, not schema_version bump

Layer B's Phase 0 Step 0.2a parser is dict-based YAML. Unknown keys are ignored naturally. v1.4.0 extends the schema with optional `<field>_source_citation` nested keys (e.g. `idea_summary_source_citation`). `schema_version` stays at `1`.

Bumping to `schema_version: 2` would force:
- Layer B Step 0.2a to branch on version (parser code change).
- A new regression surface area (pre-v1.4.0 installs read post-v1.4.0 files).
- A migration story that has no semantic value — the new keys are purely additive and optional.

Additive and silent preserves v1.3.3's zero-Layer-B-ripple discipline one version further. If Layer B ever surfaces citations in Step 0.4 card / Step 0.5 template (v1.4.1+ target), that change lands additively without a version bump too.

### Why key omission over explicit null on no-citation

When a field has no citation (image-only drop, field not present in source, fallback path fired), the `<field>_source_citation` key is **omitted** from the frontmatter rather than written as `null`.

- Fallback-path files become byte-identical (modulo `skill_version`) to v1.3.3 fixtures — the regression probe `diff` reports zero deltas modulo `created_at` + `skill_version`.
- YAML parsers consuming the file handle "key absent" identically to "key present with empty dict" — the omission form is zero-ambiguity.
- Explicit `null` adds noise without semantic value and costs 5 extra bytes per field per file.

### Why `typed_text` is wrapped in a synthetic citations-enabled document block

Citations API produces `cited_text` only for content carried in a `document` block with `citations: {enabled: true}`. Inline user text is otherwise not citeable. v1.4.0 wraps non-empty `typed_text` at index 0 of the `documents[]` array as a synthetic document (type `text`, source `typed_text`, citations enabled, 1h cache TTL). Attachments follow at indices 1..N. Image blocks are NOT wrapped — Citations API does not cite images; image blocks stay at their image content-type and carry no `citations` flag.

This makes `document_index: 0` citations in the response unambiguously refer to inline text (`text_char_range` type) when `typed_text` was non-empty; attachments start at 1. Spec § "Typed-text citation wrapping" nails the exact Python dict shape.

### Why one active model, no hardcoded Opus→Sonnet fallback

R8 § Stage 2 recommends `claude-opus-4-7` for extraction quality with Sonnet 4.6 as cost fallback. v1.4.0 defaults to Opus via a constant in the extractor, overridable via `GENESIS_DROP_ZONE_MODEL` env var. **No hardcoded Opus-then-Sonnet-on-failure logic** — a single active model per invocation keeps the extractor's control flow flat, and fallback on Opus failure routes to v1.3.3 in-context extraction (not to Sonnet). A user who wants Sonnet as primary sets the env var once. A user who wants Opus-then-Sonnet chains is out of v1.4.0 scope; ship v1.4.1+ if real pain emerges.

### Why explicit 1h cache TTL always

R8 § Stage 2 documents that Anthropic silently tightened the default prompt-cache TTL from 1h to 5m around March 2026. The extractor **always** passes `cache_control: {type: "ephemeral", ttl: "1h"}` on document blocks; never relies on defaults. The env override `GENESIS_DROP_ZONE_CACHE_TTL` accepts `5m` or `1h` for experimentation. Modification-loop cache economics depend on the 1h window — at 5m the cache breaks between card print and user response in realistic Victor-paced sessions.

### Why three fixtures, not two

- `drop_zone_intent_fixture_v1_4_0_fr_with_citations.md` — API-path happy case, FR body, citations populated. Ship-gate artefact for scenarios #28 + #35.
- `drop_zone_intent_fixture_v1_4_0_en_with_citations.md` — API-path happy case, EN body, citations populated. Ship-gate artefact for scenario #28 (EN variant) + Layer B regression.
- `drop_zone_intent_fixture_v1_4_0_fallback.md` — fallback-path file. Byte-identical to `drop_zone_intent_fixture_v1_3_3_en.md` modulo `skill_version`. Ship-gate artefact for scenarios #29 + #38.

The fallback fixture is the explicit regression probe for the byte-identity claim. Without it, a reader must infer the invariant; with it, the property is assertible via `diff`.

### Why the extractor prompt hardcodes FR canonical null-class tokens

The API receives instructions in English (extractor prompt is English). But the prompt mandates that null-class values in the output JSON are the FR canonical tokens (`"a trouver ensemble"`, `"non mentionne"`, `"non mentionnee"`, `"a affiner — X ou Y"`), regardless of the source-content language. This preserves:

1. The Layer A / Layer B contract (Layer B Step 0.2a matches on exact FR canonical strings).
2. The v1.3.3 body-vs-frontmatter asymmetry discipline (body = locale-detected human echo; frontmatter = FR canonical data contract).
3. Backward compatibility with v1.3.3 fixtures.

R9 is preserved: the prompt is English (R9 tier 1 dev-layer), but it quotes FR canonical data-layer strings (allowed — data quoting is not prose).

### Target self-rating calibration

Aspirational ≥ 9.3 on every axis; floor ≥ 9.0 on every axis. Expected strengths:

- **Pain-driven 9.1–9.3** — closes the last big structural lever for `genesis-drop-zone` (audit-trail via API-hard citations). Pain is real: v1.3.1–v1.3.3 extraction is honest but has no source attribution; an EN-native Victor looking at a mirror has no way to verify "Genesis saw this on page 1 of my brief".
- **Best-at-date 9.2–9.3** — inline R8 citations preserved; `v2_promptor_fusion_landscape_2026-04-17.md` Stage 2 still fresh through 2026-04-24. New R8 stack entry for `anthropic-python` at ship time.
- **Self-contained 8.9–9.1** — one skill touched (`genesis-drop-zone`) but new runtime surface (Python subprocess + env var + SDK dep) broader than v1.3.3. Honest accounting: MINOR bump means the ship is not as self-contained as a PATCH.
- **Anti-Frankenstein 9.1–9.3** — each element earns its place (Python for SDK; env check for pre-flight; subprocess for isolation; 1h TTL for documented API regression; key omission for byte-identity). Cross-skill-pattern #2 refinement is the single permanent structural change; all other additions are skill-local.
- **Prose cleanliness 9.1–9.3** — spec addition is substantive (+287 lines) but follows the living-spec pattern; spec polish addressed 4 load-bearing advisories. SKILL.md mirror extension follows the established 1:1 contract.

Weighted average target: **≥ 9.1/10**. Would not be shocked at 8.8 if self-contained / anti-Frankenstein take larger hits than expected. Floor: the ship does NOT go out at < 8.5.

**Step 3 — document-block assembly**:

- [ ] Function `build_documents(attachments: list, typed_text: str, cache_ttl: str) -> list` — assembles the API `documents[]` array per spec § "Typed-text citation wrapping":
  - If `typed_text` non-empty, prepend synthetic document block at index 0 (type `text`, source.type `text`, media_type `text/plain`, `citations: {enabled: True}`, `cache_control: {type: "ephemeral", ttl: cache_ttl}`, `title`, `context`).

  - Image branch: base64-encode, wrap as **image content block** (not document block — images are not citeable). No `citations` key on image blocks; no `cache_control` on images either (cache only applies to large documents).

**Step 4 — API call + citation extraction**:

- [ ] Function `call_api(client, model, documents, content_locale_hint) -> tuple(dict, dict)` — sends Messages API call with:
  - `model` = resolved from `GENESIS_DROP_ZONE_MODEL` env or payload `model` or `DEFAULT_MODEL`.
  - `max_tokens` = 2048 (extraction JSON is small; reserve buffer for citation metadata).
  - `system` = string instructing the model to extract the 9-field schema and output ONLY a JSON object with the specified keys. System prompt hardcodes the FR canonical null-class tokens as the exact strings to use for missing fields.
  - `messages` = a single user turn with content = `documents[]` (assembled above) + image blocks (if any) + one text block requesting JSON output.
- [ ] Parse response: iterate `response.content`, collect text blocks (the JSON itself) and their interleaved `citations[]` (for per-field citation resolution).
- [ ] Build per-field citation map by re-reading the JSON output + matching citations to each field by proximity (citation carried by the text content block that produced the field's value — the SDK attaches citations to the text block, not to individual string tokens; we approximate by field order in the emitted JSON).
- [ ] Return `(extracted_dict, citation_map)`.

**Step 5 — output assembly**:

- [ ] Function `build_output(extracted: dict, citations: dict, usage) -> dict` — merges the 9-field extraction + schema_version + per-field `<field>_source_citation` entries (ONLY for fields that have a citation — key omission for fields without).

  - Include `usage` object mirroring SDK's `response.usage` (input_tokens, cache_read_input_tokens, cache_creation_input_tokens, output_tokens).

**Step 6 — main function + exit code routing**:

- [ ] `def main()`:
  - Read stdin payload.
  - Check `ANTHROPIC_API_KEY` env → `sys.exit(EXIT_NO_KEY)` if missing.
