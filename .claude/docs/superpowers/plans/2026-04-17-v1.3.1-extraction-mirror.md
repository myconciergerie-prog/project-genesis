## Design decisions

### Why in-context extraction, not Path A Citations API

Anti-Frankenstein gate: introducing an Anthropic API call (even with `ANTHROPIC_API_KEY` available in Layer 0 user profile) adds a first external-dependency privilege for `genesis-drop-zone`. That privilege has **no downstream consumer** in v1.3.1 — nothing is written to disk, nothing is handed off. A privilege with no consumer is speculation. The v1.3.1 route instead upgrades the existing ack into a structured mirror using the same multimodal understanding the skill already relied on; concentrated privilege remains `none`. Path A Citations upgrade moves to v1.3.2+ deferred list, alongside the write + handoff that will actually consume the audit-trail.

### Why mirror replaces acknowledgement, not augments

Keeping v1.3.0's bullet-list ack AND adding a mirror after would produce two "here's what I see" screens — duplicative UX for Victor. The 9-field mirror table covers both "I see your inputs" (via the `Depose` / `Dropped` row) and "I understood" (via the 8 other rows) in a single surface. Replacement is simpler and cleaner.

### Why 9 fields, not 5

User picked the extended schema at brainstorming. The 4 bonus fields (`langue_detectee`, `budget_ou_contrainte`, `prive_ou_public`, `hints_techniques`) are Victor-legible (not dev-jargon labels), describe orthogonal axes not derivable from the 5 core fields, and cost nothing extra to extract once Claude is already reading the dropped content end-to-end for the core fields. Null-visible convention (three classes: `a trouver ensemble`, `non mentionne(e)`, `a affiner — X ou Y`) keeps the mirror honest when fields aren't filled.

### Why PATCH (v1.3.1), not MINOR (v1.4.0)

v1.3.x is the conversational-layer ship line. v1.3.0 opened it with welcome + bullet ack + bridge. v1.3.1 is a refinement of the same skill's same phase — same surface, same entry point, backwards-compatible extension of the ack into a structured mirror. Not a new major feature, not a breaking change, not an unrelated thematic line. PATCH is the correct semver; naming it v1.4.0 would falsely signal a thematic break.

### Why living spec (single file) instead of a new v1.3.1-dedicated spec

v2_etape_0_drop_zone.md was written 1 day ago (v1.3.0 ship session) and is small and current. Multiplying specs per version would fragment the mirror discipline (cross-skill-pattern #1) — the drift check becomes "diff N specs against M SKILL.md sections" instead of "diff 1 spec against 1 SKILL.md". The living-spec pattern (extend with `## Scope — v1.3.1 extraction`, rename the superseded section, update the 1:1 mirror map) scales cleanly. If the skill grows to dozens of versions this choice can be revisited — for now it is straightforward.

### Target self-rating calibration

Aspirational ≥9.3 on every axis; floor ≥9.0 on every axis. v1.3.0 cleared the 8.6–8.8 PATCH-cycle ceiling at Best-at-date with inline R8 citations (ceiling broken at 9.4). v1.3.1 keeps that lever by citing the same R8 entry (`v2_promptor_fusion_landscape_2026-04-17.md`, still fresh) for the 9-field schema + token-streamed reveal + null-class discipline. Pain-driven axis strongest: v1.3.1 closes the "Extraction arrivent bientôt" promise that v1.3.0 itself could not yet keep — direct user-visible delivery of a promised capability. Anti-Frankenstein axis strong: privilege stays `none`, no API introduced speculatively.

```
### Alignment and rendering rules

- Labels padded to 14 characters (left-aligned), 2-space separator, value flows right.
- **ASCII pure inside the table block** — no accents in label or value rows (v1.3.0 rule preserved for Windows code-page stability). Accents are allowed in the `◐ Je regarde...` opening line and the `✓ Lu et compris.` closing line (plain prose, routed through the UTF-8-stable stream path).
- Row reveal is **token-streamed** one-by-one. `◐` stays visible until the 9th row renders, then `✓` closes.

### Per-field rules

| Field | Label FR | Null-class when missing |
|---|---|---|
| `idea_summary` | Idee | class 1 degenerate: `a trouver ensemble` (mirror only fires when content non-zero) |
| `pour_qui` | Pour qui | `a trouver ensemble` (core) |
| `type` | Type | `a trouver ensemble` (core) |
| `nom` | Nom | `a trouver ensemble` (core) |
| `attaches` | Depose | `texte seul` if no attachment; else list 3 items + `+ N autres` beyond |
| `langue_detectee` | Langue | always filled: `FR` / `EN` / `mixte` |
| `budget_ou_contrainte` | Budget | `non mentionne` (bonus) |
| `prive_ou_public` | Visibilite | `non mentionnee` (bonus) |
| `hints_techniques` | Tech | `non mentionne` (bonus) |

### Truncation rules

- Each row value ≤ 60 chars after label; truncate at 57 + `...`.
- `Depose` lists at most 3 items; beyond 3, append `+ N autres`.
- `Idee` value: user's own words when possible; condense paragraph to ≤ 60 chars. Full content kept in Claude's context (v1.3.2+ handoff).

### Ambiguity branch

If the content supports multiple interpretations on a field:

```
   Type          a affiner — boulangerie ou restaurant
```

`a affiner` is the third null-class label alongside `a trouver ensemble` and `non mentionne(e)`.

### Zero-content branch

If the user's response contains only the trigger phrase with no content to echo:

```
 Je t'écoute — dépose ou écris ce que tu veux me partager.
```

No `◐`, no mirror, no `✓`. Skill waits for user's next turn and re-runs the mirror flow when content arrives. (v1.3.0 branch preserved unchanged.)

### Unreadable-attachment branch

If Claude cannot read an attached file (exotic binary, oversize PDF > 32 MB × 600 pages), the `Depose` row lists the file alongside readable items:

```
   Depose        1 brief "X" + 1 fichier illisible : <filename>
```

Extraction of the other 8 fields continues from readable content. Graceful, no error code.
```
