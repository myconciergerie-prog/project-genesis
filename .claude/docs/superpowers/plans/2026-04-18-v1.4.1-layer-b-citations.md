## Design decisions

### Why v1.4.1 closes Layer B citation surfacing now

v1.4.0 shipped Citations API extraction at Layer A. The `drop_zone_intent.md` frontmatter gained optional `<field>_source_citation` keys. The Layer A mirror renders `[page N]` / `[lines X-Y]` annotations inline. But `genesis-protocol` Phase 0 Step 0.4 (the intent card Victor sees in detailed/semi-auto mode) does not render these annotations — Victor sees "Vision: boulangerie artisanale pour livraison matin" without provenance, even though the provenance is one `Read` call away in the frontmatter. Closing this asymmetry is pure loop-closure: same data, one additional rendering site.

### Why PATCH (v1.4.1), not MINOR (v1.5.0)

Read-only rendering. Zero new privilege, zero new dependency, zero new subprocess surface, zero new network surface, zero schema bump, zero Layer A ripple, zero new fixture, zero new bilingual pair. Every structural-weight argument that justified MINOR for v1.4.0 is absent. The change is pure surface enrichment on Layer B. PATCH is honest.

### Why inline suffix inside Value column, not dedicated column or section

Three options considered at spec brainstorm:
- **(a) Dedicated `## Source attribution from drop zone` section** in Step 0.5 with field→citation mapping table. Rejected: introduces a second source of truth for field-value attribution that the reader must mentally reconcile with `## Fields`.
- **(b) Third `Attribution` column** in the `## Fields` table. Rejected: changes the table's structural shape (3 cols → 4) and ripples into legacy `config.txt`-sourced sessions that have nothing to put in the new column.
- **(c) Inline suffix inside existing `Value` column** — selected. Minimum-surface: one string concatenation per row, no layout change, citation lives exactly where the value lives.

### Why propagated citations for Project slug + Is-a-plugin

Project slug is **derived** from `nom` (lowercase, strip accents, alphanumeric + `-`). Is-a-plugin is **inferred** from `type` (substring match on "plugin"). Both are deterministic functions of a cited source field. The honest rendering propagates the citation from the upstream source: Project slug cites `nom_source_citation`, Is-a-plugin cites `type_source_citation`. Alternative (no propagation — just empty suffix) would hide the provenance of deterministically-derived values from Victor. Propagation is the transparent stance.

### Why Mixed media stays unadorned

The `Mixed media` card row sources its value from Step 0.3 disk `Glob` — not from Layer A's `attaches` frontmatter field. The two sources can legitimately diverge (Victor drops "logo.png" but the actual file on disk is `brand_logo.png`). Surfacing `attaches_source_citation` on the Mixed media row would cite the wrong provenance. Honest stance: `attaches_source_citation` is **preserved by Step 0.2a** (dict parse reads everything) but **not rendered** on the Mixed media row; the row stays suffix-free. Scenario #44 is the ship-gate probe for this honest-provenance rule.

### Why zero fixture churn

The v1.4.0 fixtures (`_fr_with_citations.md`, `_en_with_citations.md`, `_fallback.md`) carry all citation shapes (`pdf_page_range` + `text_char_range` + no-citations). v1.4.1 tests Layer B's render logic against the existing fixtures as regression-set artefacts. Creating new v1.4.1 fixtures for a read-only feature would be Frankenstein — the invariants we care about (citation dispatch correctness, per-row gate independence, Mixed media unadorned honesty) are already probe-able against v1.4.0 fixtures with minor synthetic variations for scenario #43 (partial citations).

### Why master.md cross-skill-pattern #4 gets one-paragraph discipline upgrade

Pattern #4 has now accumulated four data-points (v1.3.2 wire / v1.3.3 body-vs-frontmatter asymmetry / v1.4.0 additive keys / v1.4.1 additive rendering). Each data-point adds one stanza of discipline. v1.4.1 refines: "Layer B may opt-in to render additive keys read-only. Parser mechanics unchanged; rendering logic gains conditional branches on key presence." This statement — together with v1.3.3's asymmetry and v1.4.0's additive-keys discipline — forms the load-bearing pattern future Étape-skills will compose against.
