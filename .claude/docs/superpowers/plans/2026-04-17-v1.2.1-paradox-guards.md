## Design decisions

### F29 — Relocate `v1_rules.md` into the skill

The current path resolver in `skills/genesis-protocol/phase-1-rules-memory.md` Step 1.3 walks **three levels up** from `SKILL.md` to find `.claude/docs/superpowers/rules/v1_rules.md`. This is reliable in dev (`<repo>/skills/genesis-protocol/SKILL.md` → `<repo>/`) and plugin-dir install, but breaks in personal-scope install (`~/.claude/skills/genesis-protocol/SKILL.md` → `~/.claude/` which has no `.claude/docs/`).

**Chosen fix**: relocate the canonical rules file to `skills/genesis-protocol/rules/v1_rules.md` (inside the skill package). The skill is now **self-contained** — it ships its own rules template and no longer depends on a sibling directory at the plugin root. Update the path resolver to `<skill_dir>/rules/v1_rules.md` with a legacy-path fallback for backwards compatibility during transition.

**Rejected alternatives**:
- **Keep canonical + ship a copy**: two sources of truth, sync burden, Frankenstein.
- **Require a second `cp -r` at install**: fragile, invisible, bad UX — the F18 workaround already shows users don't know about it.
- **`$CLAUDE_PLUGIN_ROOT` env var**: would need to verify Claude Code actually sets it for skills loaded from `~/.claude/skills/`. Unknown. Leaves a subtle failure mode.

The repo-level `.claude/docs/superpowers/rules/v1_rules.md` is **removed**. All references (`CLAUDE.md`, `memory/MEMORY.md` indirect, `memory/master.md`) are updated to point at the new canonical location inside the skill.

### F30 — Replace literal `.git/` check with `git rev-parse --show-toplevel`

The current check at `phase-3-git-init.md` Prerequisites and Step 3.1 asks "does the target folder contain `.git/`?". This is a filename-level check that misses the **nested-repo** case: target is inside a parent git repo but has no `.git/` of its own. `git init` then silently creates a nested repo, breaking every downstream assumption.

**Chosen fix**: probe with `git -C "<target>" rev-parse --show-toplevel 2>/dev/null`:
- Exit 0 + output == target abs path → target *is* already a repo root → existing "resume partial bootstrap?" flow handles it.
- Exit 0 + output != target abs path → target is inside an existing outer repo → **HALT** with an actionable message pointing at the outer toplevel and recommending a sibling directory.
- Non-zero exit → outside any repo → proceed with `git init`.

### F23 + F27 — Step 0 structural stops

Two paired guards at `SKILL.md` Step 0 (before rendering the top-level consent card):

- **F23 target-inside-orchestrator**: compute orchestrator plugin root (three levels up from `SKILL.md`, same walk as phase-1-rules-memory.md). If target absolute path starts with orchestrator plugin root absolute path, HALT with "target path is inside the Genesis plugin's own tree — use a sibling folder outside the plugin repo (e.g. `C:/Dev/Claude_cowork/<your-project>/`)".
- **F27 self-collision slug**: after Phase 0 Step 0.2 parses the slug from `config.txt`, compare against a structural-stop list `{"project-genesis"}` and against the orchestrator's own slug from `<plugin-root>/.claude-plugin/plugin.json` when resolvable. If match, HALT with "slug collides with the Genesis orchestrator itself — pick a differentiated slug (e.g. append `-selfdogfood`, `-test`, or rename the project)".

These two checks combined make a deliberate strange-loop target impossible to bootstrap without explicit override — a future `--allow-nested-paradox` flag remains a v2 candidate and is **not** shipped in v1.2.1.

### Why no executable tests

The orchestrator is pure Markdown (Option A, per F32 deferral). The "tests" are scenario walk-throughs of the v1.2.0 friction replays — documented as verification steps in this plan, executed manually in the final task.

Current Step 3.1 (lines 25-33):

New:
```markdown
### Step 3.1 — `git init` in the target folder

**Pre-init probe** (mandatory, not optional):

```bash
outer="$(git -C "<target>" rev-parse --show-toplevel 2>/dev/null)"
target_abs="$(cd "<target>" && pwd)"
```

Three cases:

| `outer` value | Interpretation | Action |
|---|---|---|
| empty (exit non-zero) | Target is outside any git repo | Proceed with `git init -b main` |
| `$target_abs` | Target is already a repo root | Ask: "Resume partial bootstrap? (yes / abort)" — on yes, skip `git init` and advance; on abort, exit |
| any other path | Target is **nested inside** the outer repo | **STOP — do not `git init`** (see next) |

**When nested detected, halt with this message template** (fill in `<outer>` and `<target>`):

```
❌ Phase 3 Step 3.1 — nested git repository refused

Target folder:  <target>
Outer repo:     <outer>  (git rev-parse --show-toplevel reports this)

`git init` inside an existing repository would create a nested repo, which
breaks: push flow (wrong remote), PR lifecycle (wrong repo), worktree
archive, and every downstream assumption Phase 3/5.5/6 make about the
project being its own root.

Recommended fix: pick a sibling directory OUTSIDE the outer repo.
Example: instead of
    <outer>/sub/path/<target-basename>/
use
    <parent-of-outer>/<target-basename>/

If you intentionally want a nested-bootstrap workflow, that is a v2
candidate and currently not supported by the orchestrator.
```

After a clean `git init -b main`, confirm:

- `.git/` exists inside the target
- `git -C "<target>" rev-parse --show-toplevel` now returns `$target_abs`
- `git -C "<target>" config --local init.defaultBranch` returns `main` OR `git -C "<target>" branch --show-current` returns `main` after the first commit
```
