Les 4 questions qui ont guidé la session

Avant de lancer le ship, quatre questions stratégiques ont été posées. Chacune a débloqué une décision :

« Ça peut encore s'améliorer dans cette phase ? »
Marginalement, oui. Mais ce n'est plus le bon levier. La running average est à 8.54, les six skills sont stables. Le prochain palier d'amélioration ne viendra pas de polish supplémentaire — il viendra de frictions réelles lors d'une vraie exécution. On ship, puis on apprend du terrain.
« On passe Genesis v1 à travers Genesis v1 pour l'améliorer ? »
Oui — c'est le strange loop. Comme un compilateur C écrit en C qui se compile lui-même. Le dry-run v0.9 était un paper trace. La session v1.1 sera la première exécution réelle du genesis-protocol. Chaque friction capturée deviendra un patch v1.1.
« Comment Genesis s'insère au quotidien dans un projet existant ? »
Genesis est deux choses (voir ci-dessous) : un bootstrapper utilisé une seule fois par projet, et un plugin de discipline utilisé à chaque session. Les 5 skills non-orchestrateurs fonctionnent dans tous les projets dès que le plugin est installé — sans aucune action supplémentaire.
« C'est quoi le livrable concret pour mes amis beta testeurs ? »
2 commandes dans PowerShell. Zéro Chrome. Zéro VS Code.
irm https://claude.ai/install.ps1 | iex installe Claude Code.
/plugin install project-genesis@myconciergerie-prog/project-genesis installe Genesis.
Ensuite : un config.txt + une phrase trigger. Claude prend la main.

Genesis = 2 choses

L'insight clé de cette session : Genesis n'est pas qu'un outil de bootstrap. C'est un plugin de discipline qui reste actif longtemps après le setup initial.

🏗 Un bootstrapper

Utilisé une seule fois par projet

  • 7 phases automatisées
  • config.txt → projet complet
  • SSH, PAT, repo GitHub
  • Architecture mémoire
  • Règles R1-R10
  • Premier commit + tag v0.1.0

💡 Un plugin de discipline

Actif à chaque session, dans tout projet

  • Journal system (capture de pensées)
  • Session post-processor (archivage auto)
  • Pépite flagging (découvertes)
  • Research cache avec TTL
  • Discipline worktree R2.1
  • Self-rating R10.3

Pour les projets existants comme Aurum ou Cyrano, c'est le côté « plugin de discipline » qui apporte le plus de valeur immédiate. Le côté bootstrapper nécessite un mode retrofit (détecter ce qui existe déjà, combler les trous) — c'est le premier développement v1.1.

Le concept retrofit

Genesis v1.0 bootstrap des projets neufs. Mais les projets existants ont déjà git, SSH, un repo… ce qu'il leur manque, c'est l'architecture mémoire et les règles de session. Le mode retrofit (v1.1+) détecte l'existant et comble les trous :

aurum-ai
git ✓   SSH ✓   memory ✓
Manque : règles R1-R10, research cache, journal/, pepites/, sessions/
myconciergerie
À analyser
Dépend de l'état actuel. Full bootstrap ou retrofit partiel.
cyrano
Dual-path
Existe à 2 chemins (Claude_cowork + AntiGravity). Unification via Meta-Memory Layer 1.

Le dossier de départ = tout ce qui décrit ton projet

Genesis ne se limite pas à un fichier texte. Phase 0 analyse tout le contenu du dossier :

📄
config.txt
Idée, stack, objectif
📷
Photos
Claude voit les images
📎
PDFs
Menus, tarifs, briefs
🔗
URLs
Références, concurrence

Un restaurateur met son menu.pdf + les photos de ses chambres + 3 lignes de config.txt. Un prof met sa grille tarifaire + son planning. Un développeur met un cahier des charges complet. Genesis s'adapte au volume et à la richesse du seed.

Et si le user ne sait pas quoi écrire ?
Concept Promptor (v1.2) — intégrer une interview guidée dans Phase 0 quand le dossier est vide ou le config.txt trop maigre. Inspiré de Promptor, un outil de reverse prompt engineering qui aide à structurer ses idées par questions successives. Le power user qui arrive avec un config.txt complet skip tout ça automatiquement — zéro friction ajoutée.
