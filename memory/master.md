# Project Genesis — master vision

## What this project is

A Claude Code plugin that transforms a folder containing a `config.txt` seed (plus optional mixed media) into a fully bootstrapped project — rules, memory architecture, research cache, git repo, and resume prompts — via a 7-phase protocol orchestrated by Claude Code itself. The protocol minimizes user friction to the security floor of unavoidable manual interventions (logins, admin passwords, browser extension permissions, 2FA, device authorization).

The plugin ships:

- **Phase -1 Dependencies Pre-flight** (core) — 3-mode ladder (detailed / semi-auto / auto), multidevice baseline
- **Phase 5.5 Auth Pre-flight** — SSH keygen, fine-grained PAT creation, empty repo creation, 3-probe pre-flight test
- **Journal system** — stratified thought capture, 6th memory type
- **Session post-processor** — JSONL transcript redaction + Markdown archive
- **Pépite discovery flagging** — red-light system with cross-project routing, 7th memory type
- **Canonical R1-R10 rules** — adapted for any new project

## Why it exists

Born during the Aurum.ai v0_init session on 2026-04-14 as a 7-phase template markdown committed at `0b1de3d`. After Aurum v0_init, the user observed that the template's natural home was its own dedicated repo with its own lifecycle, different audience, and potentially different license. The recursive self-bootstrap (applying v0 to itself to produce v1) is dogfooding from day zero — every friction from v0 surfaces in real time during self-application, and each friction informs a v1 improvement.

The user framed it as a "strange loop" (Hofstadter). Compiler bootstrapping philosophy: the first C compiler was written in assembly, rewritten in C, and compiled by itself. Every C compiler since 1971 self-compiles. Genesis follows the same move.

## Layer 0 inheritance — how project memory defers to the universal layer

Genesis's project memory (this `memory/` directory plus `CLAUDE.md`) does **not** duplicate universal rules, user profile, hard disciplines, workflow patterns, or machine-specific reference material. Those live in **Layer 0** at `~/.claude/CLAUDE.md` and are auto-loaded into every Claude Code session on this machine — including every Genesis session. Project memory inherits them by reference and only adds Genesis-specific material on top.

Concretely, the following are **owned by Layer 0**, not by this project: the user profile (identity, language, working style, AI stack); hard rules R2.1 worktree discipline, R9 language policy, additive auth, no-new-windows, async-mid-flow; workflow patterns (per-project SSH identity, `GH_TOKEN` env override, fine-grained PAT scope checklist, Chrome profile map); the journal system spec (6th memory type, trigger phrases, stratified format); the R8 research cache convention (TTL, sota/stack/archive split); the best-practice-at-date default and anti-Frankenstein discipline. Genesis's `CLAUDE.md` at the repo root is a short pointer file that names what Layer 0 provides and what lives project-locally — it is explicitly **not** a duplicate of Layer 0.

Project-local additions Genesis **does** own: the v1 rules R1–R10 in `skills/genesis-protocol/rules/v1_rules.md` (which inherit R8 and R9 from Layer 0 by reference and add R10 plugin conventions), the seven shipped skills, the pépite discovery system (7th memory type, `memory/pepites/`), project-specific references (`memory/reference/ssh_genesis_identity.md`, `memory/reference/github_genesis_account.md`), and the scope lock rule for Aurum. ## Cross-skill patterns — the three that emerged through v0.2 → v0.8

Three patterns surfaced during skill implementation and are now named first-class conventions. They are not rules (rules live in `v1_rules.md`); they are **patterns** that rules compose into. When a future skill is added, these are the composition surface to reach for before inventing something new.

**1. 1:1 spec mirror discipline.** When a skill is a faithful implementation of a canonical source document, the skill's `SKILL.md` mirrors the source structurally — same sections, same ordering, same table rows — and drift between the two is a merge-blocker. Applied three times so far: `journal-system` mirrors Layer 0's "Journal System — Universal Thought Capture" section; `pepite-flagging` mirrors `specs/v1_pepite_discovery_flagging.md`; `genesis-protocol`'s `SKILL.md` mirrors this `master.md` 7-phase table. The discipline pays for itself: every file explicitly commits to tracking its source, the rating ceiling is predictable, and code review becomes "diff the two" rather than "read and cross-check". **v1.6.0 adds a fourth data-point**: `promptor` mirrors the Layer 0 pépite at `~/.claude/memory/layer0/pepite_promptor_template_anthropic_prompt_engineering.md` with the skill as canonical source of truth and the pépite as a synced cache for sessions outside Genesis-bootstrapped projects (inverting the read-direction of the prior three data-points where Layer 0 / specs were canonical and the skill was the implementation — here the marketplace-shipped skill is canonical, and the Layer 0 entry is the cache that propagates to non-Genesis sessions). **v1.6.1 extends this data-point in depth (no new ordinal)** — both surfaces (skill + synced-cache pépite) now carry the same SOTA-validated edits from `sota/anthropic-prompt-engineering_2026-04-19.md`, closing the 8.7/10 Best-at-date axis cap from the v1.6.0 honest rating and demonstrating that 1:1 mirror discipline survives external-canon rev-up without schema change.

**2. Concentrated-privilege map.** Every Genesis skill has **at most one concentrated privilege per operation class** (refinement landed with v1.4.0 when `genesis-drop-zone` needed an orthogonal network class alongside the v1.3.2 disk class — pre-v1.4.0 wording was "at most one concentrated privilege per skill"). An operation class is a coarse category: `disk` (any write outside the skill's own scope), `network` (any external HTTP/API call), `subprocess` (any fork/exec outside the harness), etc. Multi-class declarations are acceptable when each class ships with its own consent model, its own five mitigations, is independently disableable (one class can be opted out without losing the other), and has its own failure mode that never escalates privilege on the other class. A skill that accretes a third class should trigger a hard anti-Frankenstein review — at that point the skill probably needs splitting. The seven data points so far: `phase-minus-one` (running installers), `phase-5-5-auth-preflight` (SSH/PAT/repo creation), `journal-system` (none), `session-post-processor` (writing redacted archives, halt-on-leak mitigated), `pepite-flagging` (pointer files into sibling projects, per-target consent mitigated), `genesis-protocol` itself (writing an entire new project directory outside the Genesis repo, top-level consent card mitigated), and `genesis-drop-zone` — **first multi-class declaration in Genesis**: disk class (writes `drop_zone_intent.md` to cwd after consent card, halt-on-existing, no mkdir — v1.3.2 ship) + network class (v1.4.0 — subprocess → Anthropic Messages API for Citations-enabled extraction, `ANTHROPIC_API_KEY` pre-flight, silent graceful fallback to v1.3.3 in-context extraction on any failure class, 1h cache TTL explicit, token-budget logging to stderr). Cross-version chain: v1.3.0 surface + v1.3.1 structured extraction + v1.3.2 first Layer A disk privilege + Layer B Phase 0 handoff wire + v1.3.3 runtime locale rendering (no new privilege class) + v1.4.0 second privilege class (network) with additive `<field>_source_citation` frontmatter keys preserving zero-Layer-B-ripple + **v1.4.1: privilege map unchanged** — Layer B-only rendering extension (Step 0.4 card + Step 0.5 `bootstrap_intent.md` template render the `<field>_source_citation` keys as inline `[page N]` / `[lines X-Y]` suffix), **zero Layer A ripple** (no `genesis-drop-zone` file modified), no new privilege class. **v1.5.0 disk class extension** — the disk class privilege now covers (a) writing `drop_zone_intent.md` to cwd after consent (v1.3.2 behaviour preserved as first-write path), (b) writing `drop_zone_intent_history/v<N>_<ts>.md` archive entries on supersession, (c) overwriting `drop_zone_intent.md` with a new snapshot after archive. The five mitigations are extended: arbitration card consent gates the entire write/archive operation; `drop_zone_intent_history/` directory created as a sibling without `mkdir -p` magic; ISO8601 UTC timestamp guarantees archive filename uniqueness; Victor-exit safety = no partial state on disk; bidirectional supersession traceability via `supersedes_snapshot` + `superseded_by` pointers. **Network class fallback retired** — v1.4.0's silent graceful fallback to v1.3.3 in-context extraction is removed; failure modes (exit codes 2-7) route to bilingual halt-with-remediation card. Anti-Frankenstein retroactive (preemptive feature, never pain-driven validated; R8 research `sota/anthropic-auth-and-oauth-status_2026-04-19.md` confirms no first-party OAuth path for Messages API in April 2026 means halt-with-remediation is the only ToS-clean contract). Cross-class independence preserved: disk and network privileges remain orthogonal with their own consent models and mitigation sets. Concentrating privilege per-class is the anti-Frankenstein discipline applied to write surfaces. Naming this map makes it a precedent every future skill must answer to: what are your concentrated privileges per class, and what are their mitigations? **v1.6.0 adds promptor as the 8th skill** with privilege class `none` (joins `journal-system` as the 2nd none-class skill ; meta-tool with no disk/network/subprocess class — the cleanest possible map entry, no mitigations needed since no privilege exists).

**3. Granular-commits-inside-feat-branch.** Every feature session since v0.5 has used the same discipline: work happens in a feat (or chore) branch under `.claude/worktrees/`, each meaningful sub-step is a separate commit, the branch is pushed and a PR opened, the PR is squash-merged to main, then tagged. The feat branch is retained after merge per R2.5 as a forensic record. The commits inside the branch preserve the "why" for each step; the squash merge keeps `main` linear; the retained branch lets any future session walk the sub-steps. This is the composition of R2.1 (worktrees) + R2 (PR discipline) + the informal "one commit per idea" convention — naming it here makes it explicit so new sessions do not have to re-derive it.

**4. Layer A / Layer B stratification (v2-onwards, added v1.3.0).** Genesis v2 stratifies the bootstrap into two layered protocols: Layer A (the conversation — Étapes 0, 1, 2, 3) and Layer B (the engine — Phases -1, 0, …, 7). Layer A skills capture intent and translate it for Layer B; Layer B skills do the actual bootstrap work. `genesis-drop-zone` is the first Layer A skill, shipped v1.3.0 as a welcome vertical slice. `genesis-protocol` remains the Layer B orchestrator. The split is a composition pattern, not a rule — new skills declare which layer they live in, and cross-layer dependencies flow Layer A → Layer B, never the reverse. This pattern was not needed while Genesis v1 was an engineer-only tool; it emerged as soon as the v2 conversational surface was introduced. **v1.3.2 is the first cross-layer wire live** — `genesis-drop-zone` writes `drop_zone_intent.md` at cwd (Layer A privilege), `genesis-protocol` Phase 0 detects and consumes it (Layer B read), with precedence over legacy `config.txt`. The pattern is documented in `.claude/docs/superpowers/specs/v2_etape_0_drop_zone.md § "Layer B integration — genesis-protocol Phase 0 (v1.3.2)"` and serves as the reference implementation for future Étape 1 → Phase 1, Étape 2 → Phase 2, Étape 3 → Phase 3 wires. **v1.3.3 preserves zero-Layer-B-ripple as a deliberate design discipline** — when Layer A added runtime locale rendering to welcome + mirror + consent card + halt + bridges + body echo, the frontmatter data contract with Layer B was held intentionally unchanged (null-class tokens stay FR canonical regardless of `content_locale`). Body = locale-detected human echo; frontmatter = FR canonical data contract. This asymmetry is the v1.3.3 reference for any future Layer A rendering polish that risks rippling into Layer B — the default answer should always be "preserve the contract, polish the surface". **v1.4.0 extends the discipline one version further via additive frontmatter keys** — the Citations API extraction path writes optional `<field>_source_citation` nested entries into `drop_zone_intent.md` without bumping `schema_version` and without removing or renaming any existing key. Layer B's Step 0.2a dict-based YAML parser ignores unknown keys naturally; Layer B code remains untouched. Precedent for any future Layer A feature that could push new information into the cross-layer contract: additive frontmatter keys + zero parser change is the default route, with schema_version bump reserved for genuinely incompatible restructurings. **v1.4.1 extends the discipline one version further via opt-in additive rendering** — Layer B's Step 0.4 card and Step 0.5 `bootstrap_intent.md` template now render the `<field>_source_citation` keys inline (`[page N]` / `[pages N-M]` / `[lines X-Y]` suffix per field), without bumping `schema_version`, without growing the Step 0.2a parser, and without touching Layer A. Fourth data-point of the zero-ripple principle: v1.3.2 wire + v1.3.3 body-vs-frontmatter asymmetry + v1.4.0 additive keys + v1.4.1 additive rendering. "Ripple" is now measured at two levels — parser-level (unchanged across the v1.3.2 → v1.4.1 range) and contract-level (forward-compat with old writers: old Layer A + new Layer B = zero citations rendered since no keys exist; new Layer A + old Layer B = citation keys ignored, no crash). Future Étape-skills composing on this pattern default to "additive keys + additive read-only rendering", bumping schema_version only for genuinely incompatible restructurings. **v1.5.0 extends the discipline one version further via additive revision-state metadata** — Layer A writes three optional frontmatter keys (`snapshot_version`, `arbitrated_fields`, `supersedes_snapshot`) on first / supersession writes, plus retires v1.4.0's silent fallback in favour of explicit halt-with-remediation card (anti-Frankenstein retroactive per R8 `sota/anthropic-auth-and-oauth-status_2026-04-19.md` confirming no first-party OAuth path for Messages API in April 2026). Layer B's Step 0.4 card and Step 0.5 `bootstrap_intent.md` template render a `⚖` suffix on rows whose semantic field name appears in `arbitrated_fields`, without bumping `schema_version`, without growing the Step 0.2a parser, and without touching Layer A's write logic. **Fifth data-point of the zero-ripple principle**: v1.3.2 wire + v1.3.3 body-vs-frontmatter asymmetry + v1.4.0 additive keys + v1.4.1 additive rendering + v1.5.0 additive revision-state rendering. The v1.5.0 ship is also the first to apply the discipline to a non-provenance metadata class (revision-state vs source-attribution), confirming the pattern generalizes beyond citation surfacing. This data-point is also the **twelfth consecutive ship ≥ 9.0** for project-genesis (12/12 since v1.2.1, projected v1.5.0 self-rating ≈ 9.20-9.30). **v1.5.1 confirms the discipline holds under PATCH constraint** — the Phase 0.5 clarification + halt card collapse + v1.4.0 fallback retirement propagation touched only Layer A surfaces (`skills/genesis-drop-zone/SKILL.md` + `phase-0-welcome.md`) with zero Layer B parser, rendering, or contract changes. **Sixth data-point of the zero-ripple principle**, demonstrating the principle survives PATCH-level prose cleanup as well as MINOR-level feature additions ; now validated across both expansive (v1.3.2 → v1.5.0) and corrective (v1.5.1) ship classes. **v1.6.0 promptor ships as an orthogonal meta-skill** (neither Layer A nor Layer B, invocable cross-session/cross-project outside the bootstrap protocol). Zero edits to any Layer A or Layer B skill ; the only cross-cutting touches are this `master.md` (data-points + this orthogonal-meta-skill note) and `skills/README.md` (entry). **Seventh data-point of the zero-ripple principle**, extending the principle from "Layer A grows / corrects / opt-in renders without rippling Layer B" to "new orthogonal meta-skill ships without rippling either layer". Per anti-Frankenstein, no third "Layer M" category is introduced for one skill ; will emerge naturally if a 2nd meta-skill arrives. **Eighth data-point of the zero-ripple principle** — v1.6.1 PATCH on `promptor` ships with zero edits to any Layer A or Layer B skill (verified via 4 git-diff probes on `skills/genesis-drop-zone/`, `skills/genesis-protocol/`, other Layer B skills, `.claude-plugin/` with `plugin.json` whitelisted). Extends v1.6.0's seventh data-point "orthogonal meta-skill ships without rippling either layer" to "orthogonal meta-skill PATCH revisions stay zero-ripple under SOTA-driven template rework". **v1.6.2 depth update on the sixth data-point** (no new ordinal) — Layer A-only runtime-evidence capture session ships with zero Layer B ripple per AC10 machine check. Structurally the same ripple class as v1.5.1 Layer A-only corrections (sixth data-point) : the work differs (runtime evidence vs paper-trace / prose correction) but the ripple is identical (zero Layer B diff line). Per v1.6.1 precedent (pattern #1 depth update on fourth, no new ordinal), same-ripple-class extension is a depth update, not a new ordinal. A ninth data-point remains reserved for a genuinely new ripple mode.

These four patterns compose. A skill that adds a new privilege must extend the concentrated-privilege map. A skill that implements a canonical spec must declare its 1:1 mirror. A session that touches code must use granular commits inside a worktree branch. A skill that ships under v2-onwards conventions must declare its layer. Any future skill or session that breaks one of these four should call it out explicitly in its self-rating — prose cleanliness and anti-Frankenstein axes both take a hit otherwise.
